# Contributing to `spooder`

## Bug Reports
If you have found a bug in `spooder`, please open an issue on the [GitHub issue tracker](https://github.com/Kruithne/spooder/issues). Please try to include as much information as possible to understand and reproduce the issue.

Here are some examples of information that may be useful:
- The version of Bun you are using.
- The version of `spooder` you are using.
- System information (OS, CPU architecture, etc.)
- Code snippets that reproduce the issue.
- Error messages and/or stack traces.
- What you expected to happen and what actually happened.
- Screenshots or videos demonstrating the issue (if applicable).

## New Features
When submitting or requesting a new feature, please be as descriptive as possible about what you want to achieve and why the feature belongs in the project.

Please keep in mind that `spooder` is not designed as a general-purpose web server and is built with specific use-cases in mind. If your feature request is not in-line with the goals of the project, it may be rejected.

If you are submitting a pull request, please ensure that your code adheres to the code style guidelines below.

## Code Style
When submitting a pull request, please ensure that your code adheres to the following code style guidelines.

This style guideline is not exhaustive and is not strictly enforced, but it is recommended to follow it as closely as possible to ensure consistency within the project.

- Indentation must be using tabs, not spaces.
- Quotes must be single quotes, not double quotes.
- All lines of code should be terminated with a semi-colon.
- Single-line statements should not include scope braces.
- Function names should be in `snake_case`.
- Class names should be in `PascalCase`.
- Scope braces should be on the same line as the statement.
- Opening scope braces should be preceded by a space.
